# reckon-fnqs.3 ‚Äî codebase analysis: subject/body title convention

## Verification of "nothing in the node layer needs to change"

**True, empirically confirmed.** `internal/node/node.go` never special-cases
newlines: `Parse`/`ParseAt` stores everything from `bodyStart` to EOF verbatim
into `n.Body` (node.go:135), and `Serialize()` returns `n.Raw` unmodified
(node.go:143). A multi-line body already round-trips byte-identical today ‚Äî
built the binary and confirmed live:

```
$ rk todo add "$(printf 'Subject line here\n\nBody line 1\nBody line 2\nBody line 3')"
```
produced a file whose body is exactly `Subject line here\n\nBody line 1\nBody line 2\nBody line 3` and `rk query --raw "SELECT body FROM nodes"` echoed it back with `\n` JSON-escapes intact, matching envelope.go:18's documented guarantee ("no object ever contains a literal newline byte" in NDJSON).

Refinement to the ticket's own framing: "todo.go:341 addDurableTodo joins argv into a single line" is imprecise about *where*. The join is `body := strings.TrimSpace(strings.Join(args, " "))` at **todo.go:272** (`runTodoAddE`), not inside `addDurableTodo` (line 341 is just `node.NewNode("todo", author, body+"\n")`, using whatever `body` it's handed). `strings.Join` only inserts spaces *between* argv elements ‚Äî it does not strip or collapse `\n` bytes *within* a single argv element. So a caller who passes one shell-quoted argument containing embedded newlines (e.g. `rk todo add "$(cat <<'EOF' ... EOF)"`) already gets a multi-line body today, mechanically. What's actually missing is everything *downstream* of storage: no derived title, and the pretty printers dump the raw multi-line body inline (confirmed live ‚Äî `todo list` produces 4 physical lines for one item). There is no ergonomic first-class way to author "subject + blank + body" (no `--body` flag, no stdin path) ‚Äî flag this as [OPEN] for the planner: is CLI ergonomics for authoring in scope, or is "pass a single multi-line argv string" acceptable and out of scope? The ticket's Scope section doesn't list a CLI-authoring change, only display + index + query, so [INFERRED] it's out of scope, but the "no body path" line invites the planner to double check this reading.

## `rk query` ‚Äî what "select title without string surgery" actually requires

Two distinct code paths in query.go handle result rows differently, and only one is automatically fixed by adding the `title` column:

1. **`--raw` mode** (`rawObjects`, query.go:509-520): scans the SQL result's columns directly into JSON objects. Once `title` is a real column on the `nodes` view, `rk query --raw "SELECT id, title FROM nodes"` works immediately ‚Äî **zero code change needed**, purely a schema/view change. This alone satisfies the "Done when: rk query can select title without string surgery" line.
2. **Canonical mode** (default, no `--raw`) ‚Äî `canonicalObjects` (query.go:401-416) throws away every column from the original SELECT except `id`, then calls `reconstructEnvelopeObject` (query.go:420-462), which re-queries `SELECT ulid, type, time, author, body, loc FROM nodes WHERE id = ?` and marshals a `node.Envelope` (envelope.go:19-33). **`node.Envelope` has no `Title` field.** So `rk query "SELECT id, title FROM nodes"` (no `--raw`) silently ignores the `title` the caller asked for ‚Äî the original SELECT's column list barely matters in canonical mode except to supply `id` and drive `--fields` filtering (`applyFieldFilter`, query.go:525-541) over whatever envelope fields exist.

[OPEN] for the planner: does canonical mode need a `Title` field added to `node.Envelope` + `reconstructEnvelopeObject`'s query (envelope.go:19-33, query.go:420-438), or is `--raw` sufficient to satisfy the ticket's "Done when"? The ticket's own scope line ("query pretty output render title only") is hard to reconcile with query.go's actual behavior: `rk query` has **no separate "pretty" summary mode** at all ‚Äî `runQueryE` forces `mode = output.NDJSON` whenever the caller didn't pass `--json`/`--ndjson` (query.go:102-105), so every non-`--raw` invocation already emits full structured envelopes, never a truncated one-line-per-row summary the way `todo list`/`today`'s `Pretty()` methods do. There is nothing in query.go to "render title only" against. [INFERRED] the ticket's query bullet is really about the raw/canonical column-availability gap above, not a new query summary mode ‚Äî worth confirming with whoever filed the ticket rather than assuming.

## Existing patterns to follow / conflict to reconcile

**A different, pre-existing "title" convention already exists and will collide.** `internal/cli/note_v1.go` (`rk note create`/`rk note show`/`rk note rename`) stores title as an **explicit frontmatter prop**, not derived from the body:
- `note_v1.go:324`: `props := map[string]string{"title": title}` where `title` is the CLI's positional args, space-joined ‚Äî completely independent of the note's body text.
- `note_v1.go:427`: `note show` reads `props["title"]` back out.
- `note_v1.go:531-538`: `note rename` uses `n.SetField("title", ...)` / `n.InsertField("title", ...)` ‚Äî a real, editable frontmatter field.

This is a second, incompatible meaning of "title" for the same node concept. If the new `_nodes.title` column is unconditionally derived from `Body`'s first line (per this ticket's convention) it will diverge from ‚Äî and could confuse ‚Äî the existing `props["title"]` prop that `note show`/`note index` already surface for `type: note` nodes (note titles are typically NOT the first line of the body; they're a separate short label, and note bodies commonly start with prose, not a restated title). **[OPEN], load-bearing for the planner:** does the new derived `_nodes.title` column need to prefer `props["title"]` when present (falling back to first-body-line only when absent), or is it scoped to types that don't already have an explicit title (todo/log-entry/etc.), leaving `note` nodes' existing `props["title"]` untouched and simply not consumed by the generic `nodes.title` column? The ticket text ("A durable todo (**or any node**) should hold a multi-line body...") suggests the convention is meant to be universal, which argues for the precedence rule; but nothing in the ticket acknowledges `note_v1.go`'s existing prop-based title at all ‚Äî this is a gap in the ticket, not something already decided.

No other subject/body or first-line-as-title convention exists anywhere else in the codebase (checked `logparser.go`'s `id::`/`did::` inline markers ‚Äî those are unrelated line-stripping markers, not title derivation; checked legacy `internal/models/note.go`'s `Title` field ‚Äî gen-1 DB model, unrelated to `internal/node`).

The spec doc itself already documents `title` as a **note-specific prop**: `composable-redesign.md`'s Fields table, `props` row (line 471): `"per-type open bag (todo: state,due; note: tags,title)"`. This is spec-level confirmation of the conflict above, not just a current-code accident ‚Äî whoever plans this ticket is choosing between two spec-sanctioned meanings of "title" for the same word, and the spec doc update (previous section) needs to reconcile them explicitly, not just append a new field.

**Checked: no other list surface smears body.** Grepped every `internal/cli/*.go` for `.Body` usage and for `SELECT ... body ... FROM nodes` ‚Äî only `todo.go` (`listDurableTodos`/`listEphemeralTodos`) and `today.go` (`buildAgenda`) query body for list rendering. `rk add`'s `logAddResult.Pretty()` (add.go:79-80) never echoes body at all (path/id/day/time only). `rk log`'s other verbs are legacy gen-1 DB-journal-backed (`internal/models`/`internal/service`, not `internal/node`) and out of this ticket's scope per `AGENTS.md`'s "old code / design doc wins" caveat. So the ticket's Scope enumeration ("todo list / today / query pretty output") is a complete list of current smearing surfaces, not an arbitrary subset.

## Known pitfalls

- **Checked the index/query side for the schema-change equivalent of the "pinned JSON shape" risk ‚Äî clean.** `internal/index/AGENTS.md:82` warns "tests compare a sorted dump," which sounded like it could break on a new `nodes`-view column. Verified: `index_test.go` (e.g. `TestRebuildPopulatesViews`) asserts via `count(*)`/targeted single-column queries, never a `SELECT *`-decoded-into-a-fixed-struct dump. `query_test.go` uses `SELECT * FROM nodes` extensively but only decodes via `node.ReadNDJSON` (canonical mode, ignores unknown/absent Envelope fields) or `parseNDJSONMaps` with per-key existence checks (`TestQueryTS18_Fields`, explicit `--fields` filtering) ‚Äî no test asserts an exhaustive key set on an unfiltered row. `SchemaVersion` is compared against the live constant in `TestSchemaVersionAutoRebuild` (index_test.go:363-364), not hardcoded to `2`, so bumping it to `3` needs no test edits. Net: adding a `title` column/field is safe against the existing suite either way the `Envelope` [OPEN] above is resolved.
- **Result-struct JSON shapes are documented as "pinned"** in `todo_test.go`'s and `today_test.go`'s header comments ("do not change field names/tags without updating that pin"). Checked: no test does exact-key-set/`reflect.DeepEqual`-style whole-object comparison against these structs (grepped for `ElementsMatch`/`reflect.DeepEqual`/`JSONEq` ‚Äî none found), so **additively** adding `Title` is safe, but the pinned-contract comment blocks (todo.go:139-141, todo_test.go's header, today.go:88-92, today_test.go's header) must be updated to keep them truthful ‚Äî they are hand-maintained doc comments, not generated.
- **Schema bump triggers a full rebuild for every existing vault** (`internal/index/AGENTS.md:62-66`, `index.go`'s `Open`) ‚Äî expected and fine (the index is disposable/derived), but worth calling out in the PR since it changes `rk index`'s next-run cost, not correctness.
- **`_nodes.body` already carries the value `n.Body` verbatim, including any trailing newline** `addDurableTodo` appended (`body+"\n"`, todo.go:341) ‚Äî confirmed live (`query --raw` showed `"...Body line 3\n"`). The title-derivation helper must not assume `body` is pre-trimmed; it should skip leading blank lines defensively (a body could start with `\n\n` in a hand-authored file) and take the first *non-empty* line, matching the ticket's own wording, not just `strings.SplitN(body, "\n", 2)[0]` (which would return `""` if body itself started with a blank line).
- **`splitChecklistLines`/ephemeral todos never carry a multi-line body** ‚Äî `checklistItemRe` (todo.go:952) matches one physical line, so the whole "smearing" problem is durable-todo/note/log-entry-only. Don't invent a title concept for ephemeral items that doesn't map onto anything; see the [OPEN] item above.
- **REVIEW_PATTERNS.md** has no entries specific to node/index/CLI-output work for this ticket's shape (checked). The generally-applicable standing patterns are: wrap errors with `fmt.Errorf("...: %w", err)` context (used throughout reconcile.go/query.go already ‚Äî match that style for any new error paths), and respect `--quiet` in any new stderr/stdout writes (not directly triggered here since this ticket only adds a field to existing Print paths, not new output).
- **Fuzz/round-trip gate**: since the node-layer claim holds (nothing in `Parse`/`Serialize`/`Render` changes), `TestRoundTripIdentity`/`FuzzRoundTripIdentity` (node_test.go) should need no new fixtures ‚Äî but the new `Title` helper itself needs its own unit tests (table-driven, per repo convention ‚Äî see `internal/index/AGENTS.md` "Testing" section and node_test.go's style) covering: single-line body, blank-first-line body, body with only blank lines (empty title), CRLF body (`\r\n` ‚Äî the parser leaves `\r` in `Body` per node.go/AGENTS.md:90-91, so the helper must trim it when isolating the subject line), and a body that's empty string.
