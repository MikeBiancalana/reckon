## The problem to solve

Mike's current personal-knowledge system stores **everything in one place**: Logseq daily journals (`~/logseq/journals/YYYY_MM_DD.md`). Journal logs, notes/knowledge, AND todos/tasks are all **interspersed** in the daily journal files, managed by shell helpers (`work_system.sh`: `log`/`progress`/`win`/`fin`/`carry`/`commit`/`owed`/`todo`/`intend`) that append tagged bullets to sections (`## Intentions` / `## Wins` / `## Log`). Tags: `#carry`/`#commit`/`#owed` (open tasks), `#done`/`#resolved` (closures), `#win`/`#progress`/`#intention`. Rituals on top: `/startup` (morning gather+snapshot), `/log-day` (catch-up), `/pa` (Gilbert), `/wrap-up`, `/weekly-retro`. A separate `MEMORY.md` auto-memory (one fact per file) + a `PA/observations` page also exist.

**Mike's stated pain points (the core of the ask):**
1. **Everything's the same bucket, but they're different *types*.** Journal ≠ notes ≠ tasks. They have different lifecycles and access patterns, yet all live interspersed in journal files. It only half-works because Logseq has query mechanisms to paper over it.
2. **Dead-tag litter.** The system sprinkles `#tags` everywhere that **link to nowhere** — no real page/backlink behind them. Noise, not structure.
3. **Tasks never truly close.** A todo is "completed" by *adding a new journal entry saying it's done* — the original `#carry`/`#todo` stays in an **open** state. There's no real open→done task lifecycle; carries pile up and have to be manually reconciled.

**Mike's vision / principles for the target:**
- **Separate stores for the three types** — journal (timestamped logs), notes (knowledge, Zettelkasten-style), tasks (todos / checklists / repeating). Different things stored as different things.
- **Tasks with a real lifecycle** — they *close* (open→done), support repeating tasks and checklists; not open-forever.
- **Plain text** — Mike is firm on this: plain-text storage is excellent for LLMs (greppable, diffable, portable, model-legible). Any proposal must keep plain text.
- **No links to nowhere** — links/tags that actually resolve, or don't exist.

## What to research (three threads)

1. **OKF — Google's Open Knowledge Format.** A recently-released spec standardizing "LLM-wiki" ideas. Find and read the actual spec on the web (Mike has read it; you should too — don't rely on summaries). Understand: what it standardizes, its **typed-entity model** (this is key — it "fundamentally recognizes different types," which maps onto Mike's journal/notes/tasks separation), its plain-text storage model, and how it would serve as the backing format here.
2. **The "Karpathy brain" setup.** Jim (a colleague) has had good success using **OKF as the storage layer** for a "Karpathy brain." Research what this pattern actually is (web search — the LLM-wiki / knowledge-base pattern associated with Karpathy; note there's even an `understand-knowledge` skill in this org referencing a "Karpathy-pattern LLM wiki knowledge base"). Understand the architecture and how OKF backs it.
3. **`reckon` (this repo) — Mike's own tool.** Mike built `reckon` to do exactly the separation he wants: it tracks **todos / checklists / repeating tasks** separately from **journal-style logs** and **notes**, as a Zettelkasten-style system. **Read its design docs and the thinking behind them** — start with: `docs/reckon-spec_2026-06-15.md`, `docs/reckon-redesign_2026-06-15.md`, `docs/design/composable-redesign.md` (+ its `-assessment.md` and `-rebuttal.md`), `docs/reckon-plan_2025-12-22.md`, `plan.md`, `README.md`, `QUICKSTART.md`, `AGENTS.md`, `codebase-analysis.md`.
