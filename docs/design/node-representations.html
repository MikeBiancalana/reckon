One Node, Many Representations

“Representation is the essence of programming.” — Fred Brooks, The Mythical Man-Month

A single reckon node lives in four shapes at once. Each is optimized for a different job; the same facts are arranged differently in each. Hover any colored token to light up the same datum everywhere it appears.

datum: identity (ULID) scalar meta (type/time/author) body text aliases props links / edges fragments index/file metadata

The transforms — how the shapes relate

disk .md
source of truth — Parse → node.Node
in-memory aggregate ← Serialize / Render — disk .md
node.Node — Envelope → NDJSON
interchange / wire — reconcile / insert → SQLite index
normalized graph — reconstruct → NDJSON
query output

Every operation in the system is one of these arrows. Raw bytes are authoritative; every other shape is derived from them (and rebuildable).

Where does each datum live? — the representation matrix

The heart of it. Rows are data; columns are representations. This makes the duplication (highlighted) and the one gap impossible to miss.

datum Disk .md
truth		 Node (mem)
denormalized		 NDJSON
wire		 SQLite index
normalized
identity / ULID frontmatter id: .ULID (+ .frontmatter[id]) ulid _nodes.ulid + node_key
body text ×5 after --- .Raw[bodySpan:] and .Body body _nodes.body and fts_search.body
type / time / author frontmatter keys .Type/.Time/.Author same keys _nodes columns
aliases aliases: [..] .Aliases[] aliases[] _aliases (one row each)
props (state, scheduled) scalar frontmatter .Props{} props{} _props (row per key)
links / edges ref-prop depends: + body [[..]] .Links[] (rel = key or "references") links[] _edges (raw dst + resolved dst_key)
fragments (^p3) gap ^anchor in body .Fragments[] fragments[] ✗ not stored
raw bytes ✓ the file .Raw (authoritative) ✗ not sent ✗ re-read from disk via loc
byte spans / offsets — (implicit) .fieldSpans / .bodySpan ✗ ✗ recomputed on parse
node_key (surrogate id) — (derived: ULID or file:path) ✗ ✗ _nodes.node_key (PK)
hash / mtime file metadata ✗ hash/mtime (derived) _nodes and _file_meta
The gap: fragments are parsed into the Node and carried in the NDJSON envelope, but never persisted to the index (no _fragments table; insertNode skips them). Harmless today — ULID#frag link resolution isn't wired yet — but the index will need them once block-links matter.
The duplication: the body exists in up to five places at once — the disk file, Node.Raw, Node.Body, _nodes.body, and fts_search.body. Two are removable: Node.Body is derivable from Raw+bodySpan; the _nodes.body/fts_search.body pair could collapse via FTS5 external-content. The rest are deliberate (truth vs disposable index).

Why two persistent shapes? — denormalized vs normalized

Aggregate (Node / envelope)

One self-contained object: all of a node's facts in a single struct / single JSON line. Optimized for moving a whole node around — parse it, edit it, pipe it, hand it to an agent.

  • fast to read/write as a unit
  • no joins to see the whole thing
  • poor for "all nodes linking to X" or full-text — you'd scan every object

Property graph (SQLite)

The same facts decomposed into relations: _nodes + _edges + _props + _aliases + fts_search. Optimized for asking questions across many nodes.

  • graph traversal (edges), filtering (props), search (fts) are cheap
  • rk query’s reconstructEnvelopeObject re-assembles rows back into an aggregate envelope on the way out
  • disposable: rebuilt from the disk truth, never synced

Same information, two shapes. The choice of shape is the engineering — exactly Brooks’s point.

The bridge: byte spans make edits surgical

What lets the in-memory view and the authoritative bytes stay in sync without ever regenerating the file: the parser records the byte offset of each field’s value. An edit splices only that range.

Raw (the authoritative bytes), with fieldSpans["scheduled"] highlighted:

…\n scheduled: 2026-07-02\n depends: "[[01J9Z2QH8M]]"\n …

SetField("scheduled","2026-07-09") = copy Raw[:span.Start] + "2026-07-09" + Raw[span.End:] → re-parse. Every other byte preserved; the typed view refreshes.

This is why serialize(parse(f)) == f holds by construction, and why a tool and a human can edit the same file without clobbering — the create/edit keystone.
